## Context

当前 Key 模型包含限额相关字段 (`RateLimit`, `Quota`, `UsedQuota`, `UsedCount`)，但这些字段：
- `UsedQuota` 从未被更新，限额检查失效
- `RateLimit` 完全未实现
- `UsedCount` 和使用统计在 Key 列表中查询，与 Usage 统计页面重复

这是一个 Go + Gin + GORM 项目，使用 SQLite 数据库。

## Goals / Non-Goals

**Goals:**
- 移除无效的限额相关代码和字段
- 简化 Key 模型和 API
- 保持 Key 的基本功能（过期时间、启用/禁用）

**Non-Goals:**
- 不重新实现限额系统
- 不影响 UsageLog 和统计页面功能

## Decisions

### 1. 数据库迁移策略

**决定**: 使用 GORM AutoMigrate 自动处理列删除

GORM 在 SQLite 下不会自动删除列，但会忽略不存在的字段。由于这是开发阶段的小项目，选择：
- 直接删除模型字段
- 保留数据库中的旧列（不影响功能）
- 未来如需清理，可手动执行 SQL

**替代方案**: 编写显式迁移脚本删除列 — 拒绝，因为 SQLite 删除列需要重建表，对于小项目过于复杂

### 2. API 兼容性

**决定**: 直接移除字段，不做版本兼容

这是内部项目，前端与后端同步更新。移除字段后，前端同步修改即可。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 数据库旧列残留 | 不影响功能，可接受 |
| 前端显示异常 | 同步更新前端代码 |

## Migration Plan

1. 后端：删除模型字段和业务逻辑
2. 后端：移除 API 参数和响应字段
3. 前端：移除相关 UI 元素
4. 部署：重启服务即可，无需数据迁移
