## Context

本项目是一个大模型 API 代理服务，核心目标是将多个厂商的大模型 API 统一为 OpenAI 兼容格式，并提供 Web 控制面板进行管理。

**当前状态**：全新项目，无现有代码库。

**约束**：
- 必须支持私有化部署
- 数据库使用 SQLite（可扩展支持 PostgreSQL）
- 前端必须支持中英文双语
- 必须支持流式响应（SSE）

**技术栈**：
- 后端：Go 1.21+, Gin, GORM, SQLite, gin-sessions
- 前端：Vue 3, TypeScript, Element Plus, ECharts, vue-i18n

## Goals / Non-Goals

**Goals:**
- 实现 OpenAI 兼容的 API 代理，支持 `/v1/chat/completions` 等接口
- 支持配置多个后端厂商（OpenAI、Anthropic、OpenAI 兼容厂商）
- 实现请求/响应格式转换（OpenAI ↔ Anthropic）
- 支持流式响应（SSE）的转换和转发
- 提供可视化的 Web 控制面板
- 支持模型映射和负载均衡
- 支持访问密钥管理和用量统计
- 支持用户认证和中英文双语

**Non-Goals:**
- 不支持 Anthropic 格式的客户端输入（只接受 OpenAI 格式）
- 不实现模型训练或微调
- 不提供模型市场或计费系统
- 不支持嵌入式向量数据库

## Decisions

### 1. 单向格式统一（OpenAI 格式作为统一入口）

**决策**：只接受 OpenAI 格式的客户端请求，后端支持多种厂商格式。

**备选方案**：
- A. 双向兼容（接受 OpenAI 和 Anthropic 格式）- 复杂度增加约 90%，收益有限
- B. 单向统一（只接受 OpenAI 格式）- 复杂度适中，覆盖 90%+ 场景

**理由**：绝大多数客户端和 SDK 已支持 OpenAI 格式，单向统一足够覆盖主要场景。

### 2. 数据库选择 SQLite

**决策**：使用 SQLite 作为默认数据库。

**备选方案**：
- PostgreSQL：生产级，但部署复杂度高
- MySQL：介于两者之间

**理由**：
- 项目定位为私有化部署，单文件数据库部署简单
- 通过 GORM 抽象，后续可扩展支持其他数据库
- SQLite 性能足够应对中小规模使用

### 3. 模型数据独立存储

**决策**：厂商模型信息存储在独立的 `provider_models` 表，支持同步和变更检测。

**理由**：
- 厂商模型列表会变化（新增、下线、参数调整）
- 需要追踪模型状态（是否可用）
- 支持价格、上下文窗口等元数据
- 为模型映射提供数据基础

### 4. 流式响应逐事件转换

**决策**：实现 SSE 事件级别的转换器，而非缓冲整个响应后转换。

**理由**：
- 流式响应对用户体验至关重要
- OpenAI 和 Anthropic 的 SSE 格式完全不同，需要逐事件转换
- 实现复杂度较高，但可以通过抽象 Transformer 接口管理

### 5. Session 认证 + 简单用户模型

**决策**：使用 Session 进行认证，用户表只包含基本字段。

**理由**：
- 私有化部署场景，用户数量有限，单实例部署
- Session 实现简单，无需客户端存储 token
- 不需要复杂的权限系统（admin/viewer 两种角色即可）
- 配合 Cookie HttpOnly 更加安全

### 6. 前端嵌入后端

**决策**：前端构建产物嵌入 Go 二进制文件，单进程部署。

**备选方案**：
- 前后端分离部署 - 部署复杂度高
- 前端嵌入后端 - 单二进制文件，部署简单

**理由**：简化部署体验，一个二进制文件即可运行完整服务。

## Risks / Trade-offs

### Risk 1: SSE 流式转换复杂性
- **风险**：流式响应的事件格式差异大，转换逻辑复杂，可能引入 bug
- **缓解**：
  - 实现完整的单元测试覆盖各种事件类型
  - 提供调试模式，输出原始和转换后的事件
  - 参考成熟的 LiteLLM 实现

### Risk 2: 厂商 API 变更
- **风险**：厂商可能更新 API 格式，导致代理失效
- **缓解**：
  - 模型同步机制检测变化
  - 版本化的转换器，支持渐进式更新
  - 监控和告警机制

### Risk 3: Tool Calling 格式差异
- **风险**：各厂商的函数调用格式差异较大，转换可能丢失信息
- **缓解**：
  - 详细测试各厂商的 tool calling 场景
  - 文档记录已知限制
  - 对于不支持的功能，返回清晰的错误信息

### Risk 4: 性能瓶颈
- **风险**：SQLite 在高并发下可能成为瓶颈
- **缓解**：
  - 通过 GORM 抽象，预留 PostgreSQL 迁移路径
  - 实现连接池和合理的超时控制
  - 监控数据库性能指标

## Open Questions

1. **是否需要支持 Embeddings API？**
   - 暂时纳入 scope，但优先级较低
   - Embeddings 格式相对简单，各厂商差异小

2. **是否需要支持图片生成 API（DALL-E 等）？**
   - 暂不纳入 scope，可作为后续扩展

3. **定时同步模型的默认周期？**
   - 建议默认每天同步一次
   - 可在配置中调整
