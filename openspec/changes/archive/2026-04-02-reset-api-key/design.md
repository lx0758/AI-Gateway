## Context

当前系统已有完整的 API Key 管理功能，包括：
- 创建 API Key（`POST /api-keys`）
- 更新 API Key 属性（`PUT /api-keys/:id`）
- 删除 API Key（`DELETE /api-keys/:id`）
- 查询 API Key 列表（`GET /api-keys`）

Key 模型包含：ID、Key（唯一索引）、Name、Enabled、ExpiresAt、CreatedAt、Models 关联。

认证中间件通过查询数据库验证 Key 值（`key = ? AND enabled = ?`），支持过期时间检查。

系统使用 Gin 框架，数据持久化使用 Gorm。

## Goals / Non-Goals

**Goals:**
- 提供 API Key 重置功能，生成新的 Key 值并保留所有原有配置
- 重置操作原子性，确保旧 Key 立即失效，新 Key 立即可用
- 与现有 API Key 管理流程无缝集成
- 支持审计追踪（如果有审计系统）

**Non-Goals:**
- 不改变现有认证流程
- 不修改数据库 schema
- 不实现密钥版本管理或历史记录功能
- 不处理前端 UI 实现（仅定义 API）

## Decisions

### 1. API 设计
**决策**: 使用专用端点 `POST /api-keys/:id/reset` 而非在 `PUT /api-keys/:id` 中添加 reset 字段。

**理由**:
- 重置是安全敏感操作，应该独立端点便于权限控制和审计
- PUT 用于更新属性，重置属于操作类行为，语义更清晰
- 避免与现有 Update 请求结构混淆
- 参考业界实践（如 GitHub token regeneration）

**备选方案**: 在 Update 请求中添加 `reset_key: boolean` 字段
- 缺点：权限控制粒度不够细，审计追踪不明确

### 2. Key 值生成策略
**决策**: 使用与创建 API Key 相同的生成算法（`generateAPIKey()`）。

**理由**:
- 保持一致性，避免引入新的密钥格式
- 现有算法使用 `crypto/rand` 生成 24 字节随机数，安全强度足够
- 无需修改认证中间件验证逻辑

**备选方案**: 使用 UUID 或其他密钥格式
- 缺点：需要修改认证逻辑，增加维护成本

### 3. 响应格式
**决策**: 返回格式与 `POST /api-keys` 相同，包含 masked key 和 raw_key。

**理由**:
- 一致的 API 响应格式，降低用户学习成本
- raw_key 仅在创建/重置时返回一次，之后只能看到 masked key
- 用户可以立即使用新 Key，无需额外查询

**备选方案**: 仅返回成功状态，用户需查询 API Key 详情获取新 Key
- 缺点：用户体验差，无法立即获得可用的 raw_key

### 4. 失效时机
**决策**: 重置后旧 Key 立即失效，数据库更新后即刻生效。

**理由**:
- 安全优先，最大限度减少泄露窗口
- 认证中间件每次请求查询数据库，无缓存或延迟
- Gorm 的 Updates 操作立即提交到数据库

**备选方案**: 支持旧 Key 缓存一段时间（如 5 分钟）
- 缺点：增加安全风险，违背重置目的

## Risks / Trade-offs

**风险**: 用户丢失新的 raw_key 后无法找回
- **缓解**: 提示用户立即保存 raw_key，或在返回响应时显示警告信息

**风险**: 重置期间正在使用旧 Key 的请求会失败
- **缓解**: 这是预期行为，用户应该知道重置会导致旧 Key 失效；可以考虑在重置前显示确认对话框（前端责任）

**风险**: 恶意用户通过重置功能消耗数据库资源
- **缓解**: 添加操作频率限制（如每分钟最多 5 次），或需要管理员权限（取决于业务需求）

**权衡**: 不保存 Key 历史记录
- **理由**: 简化实现，减少存储成本；如果需要审计，可以在外部审计系统记录操作日志
